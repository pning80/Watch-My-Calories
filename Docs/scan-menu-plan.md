# Scan Menu Feature ‚Äî Implementation Plan

## User Prompts

I‚Äôd like to add a scan restaurant menu feature. When the user taps the scan tab, the app should bring up two menu items right on top of it. One is "Scan Food", and the other is "Scan Menu". When the "Scan Food" option is selected, the app should assume the user will take a picture of food items and send it to Gemini for analysis. When the "Scan Menu" option is selected, the app should assume the user will take a picture of a restaurant menu and ask Gemini to recognize the dishes on the menu and estimate the calorie amounts of these dishes. The "Scan Food" flow should just use the current scan flow without any changes. Please design the "Scan Menu" flow, the prompt for Gemini and the UI screens. No need to record the meals from the menu as meals consumed by the user; the calorie amounts are just for the user‚Äôs awareness. Please iterate over your plan to optimize for good user experience. Also recognize the possibility that the scanned result is not a restaurant menu. Please use the device location in your prompt to help Gemini know what restaurant this could be.

Work on the iOS app only for now. When the iOS app is mature, we will create an Android app based on the iOS version. For scanning menu, let's do one photo each time. Keep the review step. When we wait for the response from the backend, let also show the native ad.

After the scan menu analysis is done, let‚Äôs return to today screen. Let‚Äôs also save the scanned menus, but not as consumed food on Today screen or on History. Let's design a place to store the scanned menus. Each scan should save the photo and the analysis results, only when the photo is recognized as a restaurant menu.

We should add a menu that is consistently accessible on all screens except for the scan screen. Please leverage the best practices for iOS apps. The scanned restaurant menu results should be included in this app menu. Also Settings, History, and others you think the user may need. Please note the "+" button on Today's screen. You may need to optimize the entire screen and find the right place for the "+" button.

**The following design was generated by previous planning efforts. Some of them are no longer accurate. Please use it as a reference and improve it as needed.** When you update the design, please only update the content below this point and save the user inputs above this point.

## Design Decisions

Key decisions made during the planning process:

1. **Separate tabs for Scan Food and Scan Menu** (instead of a mode toggle on the camera screen). This eliminates changes to the existing `CameraView.swift` entirely ‚Äî zero regression risk on the Scan Food flow. The tab bar acts as the mode selector.
2. **4 tabs** (Today, Scan Food, Scan Menu, History). Sits within Apple HIG's recommended 3‚Äì5 range. Settings moves to the app menu.
3. **History stays as a tab** (not in the app menu). Apple HIG says toolbar menus should hold *actions*, not *navigation destinations*. History is a daily-use feature in a calorie tracker and belongs in the tab bar. The app menu holds secondary features: Scanned Menus, About, and Settings.
4. **"+" button moves from the custom header to the navigation toolbar**, coexisting with the app menu button ‚Äî both trailing, following the Apple Notes pattern (primary action rightmost, menu to its left).
5. **About screen separates app info from Settings.** Settings focuses on things the user *configures* (profile, theme, units, AI consent). About focuses on things the user *reads* (version, legal links, support, AI disclaimer). Privacy Policy and Support links move from Settings to About to avoid duplication.

---

## Context

Users want to check calorie estimates for restaurant menu items before ordering. Currently the camera tab only supports scanning food photos. This feature adds a dedicated "Scan Menu" tab that lets users photograph a restaurant menu, sends it to Gemini with location context, and displays estimated calories per dish. Results are informational only ‚Äî nothing is saved to the meal log, but recognized menu scans are persisted for later viewing.

Alongside the feature, the app's navigation is restructured: Settings moves from a tab to the app menu, and a new Scan Menu tab takes its place. A persistent app menu (ellipsis button) provides access to Scanned Menus, About, and Settings from non-scan screens. An About screen consolidates app info, legal links, support, and an AI disclaimer ‚Äî cleanly separated from Settings which focuses on user preferences.

## Scope

**iOS only.** Single photo per scan. Keep the photo review step (Retake / Analyze Menu). Show native ad during loading. Save recognized menus (photo + results). Navigate to Today screen after done. Restructure tabs: replace Settings tab with Scan Menu tab, add persistent app menu with Scanned Menus, About, and Settings.

## 2. Scan Menu Camera ‚Äî new `MenuCameraView.swift`

A simplified camera view for the Scan Menu flow. Reuses the existing `CameraManager` and `CameraPreview` (UIViewRepresentable) for the live viewfinder ‚Äî only the surrounding UI differs from `CameraView`.

### Differences from CameraView

| Aspect | CameraView (Scan Food) | MenuCameraView (Scan Menu) |
|--------|----------------------|---------------------------|
| Meal type picker | Yes (auto-assigned by time) | No |
| "Use" button text | "Use" | "Analyze Menu" |
| Photo library support | Yes | No (camera only) |
| Multi-photo | Supported via `capturedImages` | Single photo only |
| Post-capture destination | `EstimationReviewView` | `MenuAnalysisView` |

### Layout

- **Live preview**: Same camera preview + gradient overlays as `CameraView`
- **Shutter button**: Same circular design (80pt outer, 68pt inner, `cwAccent` fill)
- **Photo review**: Full-screen captured photo with two buttons at bottom:

  - "Analyze Menu" ‚Äî same style as `CameraView`'s "Use" button (capsule, `cwAccent` fill)
- **No meal type picker** in review state
- **Camera denied state**: Same permission-error UI pattern as `CameraView`

### Location permission

On first appearance of `MenuCameraView`, request `CLLocationManager.requestWhenInUseAuthorization()` if not already determined. This pre-emptively triggers the permission dialog while the user is still on the camera screen, avoiding a jarring dialog during analysis loading.

### MenuCameraRootView (in ContentView.swift or MenuCameraView.swift)

Wraps `MenuCameraView` in a `NavigationStack`, analogous to `CameraRootView`:

- `@StateObject var cameraManager = CameraManager()`
- `@State var menuReviewData: Data?` (captured image data)
- `.navigationDestination` for `MenuAnalysisView`, keyed on `menuReviewData`
- On "Analyze Menu": set `menuReviewData` to navigate
- "Done" from `MenuAnalysisView`: set `selectedTab = .dashboard` to return to Today
- "Scan Again" from `MenuAnalysisView`: clear `menuReviewData` to return to camera
- "Try Again" from not-a-menu state: clear `menuReviewData` to return to camera

## 4. Scan Menu Flow

1. User taps Scan Menu tab ‚Üí camera preview appears, location permission requested if first time
2. Takes photo of menu
3. Photo review: "Retake" / "Analyze Menu" (no meal type picker)
4. "Analyze Menu" triggers navigation to `MenuAnalysisView`
5. `MenuAnalysisView`: shows native ad + loading spinner ‚Üí fetches location (‚â§3s timeout) ‚Üí calls Gemini
6. **Success**: auto-saves menu scan, displays results
   - "Done" ‚Üí navigates to Today tab
   - "Scan Again" ‚Üí returns to camera in Scan Menu tab
7. **Not a menu**: friendly message with "Try Again" (returns to camera)
8. **API error**: error with retry option

### "Scan Food Instead" ‚Äî removed

Since Scan Food and Scan Menu are separate tabs, there's no mode to switch. The not-a-menu state offers "Try Again" (retake in Scan Menu) and the user can always tap the Scan Food tab directly. This simplifies the flow.

## 5. Location Services ‚Äî new `LocationManager.swift`

- `CLLocationManager` with `desiredAccuracy = kCLLocationAccuracyReduced` (coarse location ‚Äî sufficient for restaurant identification)
- Async method: `func getCurrentLocation() async -> (location: CLLocation?, locality: String?)`
- Reverse geocode via `CLGeocoder().reverseGeocodeLocation()` to extract locality/city name
- Add `NSLocationWhenInUseUsageDescription` to `project.yml`: *"Your location helps identify the restaurant when scanning menus."*
- **If denied or unavailable**: proceed without location ‚Äî the prompt still works, just with less context

### Permission timing

- Request `requestWhenInUseAuthorization()` on first appearance of `MenuCameraView` (not during analysis loading)
- This avoids a jarring permission dialog appearing over the loading/ad screen
- If denied, Scan Menu still works without location context

### Fetch sequencing

- Location must be fetched BEFORE the Gemini API call (included in the prompt)
- Flow: show loading + ad ‚Üí fetch location (‚â§3s timeout) ‚Üí send to Gemini with location ‚Üí show results
- If timeout: proceed without location rather than blocking

## 6. Gemini Prompt (Menu Analysis)

```
Analyze this photo of a restaurant menu. Identify the dishes listed and estimate the calorie content for each based on typical serving sizes.

{if location available}
The user is located near {locality} (approximately {lat}, {lng}). Use this to help identify the restaurant and cuisine style for more accurate estimates.
{end if}

{unitInstruction ‚Äî same metric/US customary logic as existing food scan in Services.swift}

A restaurant menu includes printed menus, chalkboard specials, digital menu displays, drink lists, and similar documents listing food or drink items offered by a food service establishment. Receipts, grocery lists, nutrition labels, and non-food documents are NOT menus.

If the image does NOT appear to be a restaurant menu, respond with ONLY:
{"error": "not_a_menu"}

Otherwise, return ONLY a raw JSON object (no markdown, no code blocks):
{
  "restaurantName": "Name if visible or identifiable, otherwise null",

    {
      "name": "Dish Name",

      "protein": 30,
      "carbs": 50,
      "fat": 15
    }
  ]
}
```

### Changes to `Services.swift`

- Add `MenuAnalysisResult` and `MenuItemResult` structs (Codable) ‚Äî see Data Models section
- Add `func analyzeMenu(image: Data, location: CLLocation?, locality: String?) async throws -> MenuAnalysisResult`

The existing `EstimationService` protocol can be extended, or `analyzeMenu` can be added directly to `GeminiService` since no mock is needed for menu analysis yet.

## 9. Persistence ‚Äî Saving Menu Scans

### New SwiftData model `MenuScan` (in `DataModels.swift`)

```swift
@Model
final class MenuScan {
    var id: UUID
    var restaurantName: String?
    var imageID: UUID?
    var timestamp: Date
    var itemsData: Data  // JSON-encoded [MenuItemResult]

    var items: [MenuItemResult] {
        (try? JSONDecoder().decode([MenuItemResult].self, from: itemsData)) ?? []
    }

    init(restaurantName: String?, imageID: UUID?, timestamp: Date, items: [MenuItemResult]) {
        self.id = UUID()
        self.restaurantName = restaurantName
        self.imageID = imageID
        self.timestamp = timestamp
        self.itemsData = (try? JSONEncoder().encode(items)) ?? Data()
    }
}
```

Items stored as JSON-encoded `Data` ‚Äî menu items are always loaded together with their scan and don't need independent queries.

### Save flow (in `MenuAnalysisView`)

On successful analysis (items returned, no `not_a_menu` error):
1. Save photo via `ImageStorage.shared.save(imageData, id: imageID)`
2. Create `MenuScan` with restaurant name, image UUID, timestamp, JSON-encoded items
3. Insert into `modelContext`

### Schema registration (`WatchMyCaloriesApp.swift`)

```swift
let schema = Schema([UserProfile.self, FoodEntry.self, MenuScan.self])
```
